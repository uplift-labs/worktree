## Context

This repository provides git worktree isolation and cleanup for OpenCode sessions. The public CLI contract lives in `CONTRACT.md`; source code lives in root `core/` and `adapters/opencode/`; installed mirrors under `.opencode/worktree/` and `.opencode/` are generated by `install.ts` for dogfooding.

The initial OpenSpec baseline must document actual behavior rather than propose new behavior. The specs are therefore derived from the contract, implementation, tests, and repository instructions.

## Goals / Non-Goals

**Goals:**

- Capture the current behavior in domain-oriented OpenSpec capabilities.
- Preserve `CONTRACT.md` as the public CLI source of truth.
- Make future behavior changes reviewable through OpenSpec deltas before code changes.
- Keep baseline requirements testable through scenarios that map to existing or expected tests.

**Non-Goals:**

- Change command flags, output, exit codes, marker formats, hooks, OpenCode plugin behavior, or installer behavior.
- Replace `CONTRACT.md`, `AGENTS.md`, `README.md`, or tests.
- Specify private helper implementation details unless they affect observable behavior.

## Decisions

- Use one OpenSpec capability per behavior domain instead of one large spec. This keeps future changes small and lets lifecycle, installer, CLI, adapter, and governance rules evolve independently.
- Treat `CONTRACT.md` as authoritative for public CLI behavior. If a future OpenSpec change conflicts with it, the change must also update `CONTRACT.md` and tests.
- Include adapter and installer requirements because they are user-visible even though they are not `core/cmd` public CLI commands.
- Use OpenSpec archive flow for the baseline so the first source-of-truth specs are produced by a normal change lifecycle.

## Risks / Trade-offs

- Baseline drift risk -> Mitigation: validate specs now and require future non-trivial behavior changes to update OpenSpec alongside tests and contracts.
- Over-documenting internals risk -> Mitigation: requirements describe observable behavior and safety invariants, not every helper function.
- Duplicate source-of-truth risk -> Mitigation: explicitly keep `CONTRACT.md` authoritative for public CLI contract and use OpenSpec as the planning/spec layer.

## Migration Plan

- Create the baseline specs as an OpenSpec change.
- Validate the change with OpenSpec.
- Archive the change into `openspec/specs/` after validation.
- Run `openspec validate --all --strict` and repository verification.

## Open Questions

- Whether future releases should require an OpenSpec change for every `CONTRACT.md` edit.
- Whether CI should add `openspec validate --all --strict` after the baseline is accepted.
